## 概要

Claude Code v2.1.185 では、API レスポンスの待機中に表示されるストリームストールヒントのメッセージ文言とトリガータイミングが改善された。従来の「No response from API - Retrying in ...」という誤解を招きやすい表現が「Waiting for API response - will retry in ...」に変更され、表示までの待機時間も 10 秒から 20 秒に延長された。これにより、ユーザーが接続エラーと誤認するケースが減少し、より正確な状況伝達が実現される。

## 詳細

### 背景

Claude Code では、API へのリクエスト送信後にレスポンスのストリーミングが一定時間停止した場合、ユーザーに状況を通知するヒントメッセージが表示される仕組みが存在する。しかし、従来の実装にはいくつかの課題があった。

1. **メッセージの不正確さ**: 「No response from API」(API からの応答なし) という表現は、接続が完全に失敗したかのような印象を与えていた。実際にはリクエストはまだ処理中であり、単にレスポンスの到着を待機している状態に過ぎない。

2. **トリガータイミングの早さ**: 10 秒という閾値は、複雑なリクエストや回線速度が遅い環境では短すぎ、正常に処理中であるにもかかわらずヒントが表示される「偽陽性」が頻発していた。

### 主な変更点

1. **ヒントメッセージの文言変更**:
   - 変更前: `No response from API - Retrying in ...`
   - 変更後: `Waiting for API response - will retry in ...`

2. **トリガー閾値の変更**:
   - 変更前: 10 秒間の無応答で表示
   - 変更後: 20 秒間の無応答で表示

### 技術的な詳細

ストリームストールヒントは、API からのストリーミングレスポンスにおいてチャンクの到着が途絶えた場合に発動する UX メカニズムである。

**メッセージの意味的な改善点**:

- 「No response」(応答なし) は接続断絶を示唆する否定的な表現であり、ユーザーに不安を与える
- 「Waiting for API response」(API レスポンスを待機中) は、システムが正常に動作しており待機状態にあることを正確に伝える
- 「Retrying in」(リトライ中) よりも「will retry in」(リトライ予定) の方が、まだリトライしていないことを明確に示す

**閾値変更の根拠**:

- 複雑なプロンプト (大規模なコードベースの分析、多数のツール呼び出しを伴うタスク等) では、モデルの初回レスポンスまでに 10 秒以上かかることがある
- 20 秒に延長することで、正常な処理中に不要なヒントが表示される頻度が低減される
- ネットワーク遅延が大きい環境 (VPN 経由、地理的に遠いリージョンへの接続等) でも安定した体験が提供される

## 開発者への影響

### 対象

- Claude Code を日常的に使用するすべての開発者
- ネットワーク環境が不安定な状況で作業する開発者
- 大規模なコードベースや複雑なタスクを Claude Code で処理する開発者

### 必要なアクション

1. **アップデートの実施**: 以下のコマンドで最新版に更新する。
   ```bash
   claude update
   ```

2. **特別な対応は不要**: メッセージ文言と閾値の変更のみであり、設定変更や移行作業は不要である。

### 移行ガイド (該当する場合)

本リリースは内部的な UX 改善のみであり、破壊的な変更は含まれない。移行作業は不要である。

ストリームストールヒントのタイミングや文言に依存した自動化スクリプト (出力のパース等) を使用している場合は、新しい文言に合わせて更新が必要となる可能性がある。

## コード例

```
# 変更前 (v2.1.184 以前)
# 10 秒間 API からの応答がない場合に表示:
No response from API · Retrying in 5s

# 変更後 (v2.1.185)
# 20 秒間 API からの応答がない場合に表示:
Waiting for API response · will retry in 5s
```

## まとめ

Claude Code v2.1.185 は、ストリームストールヒントの UX を改善する小規模ながら重要なリリースである。メッセージの文言を「No response from API」から「Waiting for API response」に変更することで、ユーザーに対してシステムが正常に動作中であることをより正確に伝達できるようになった。また、トリガー閾値を 10 秒から 20 秒に延長することで、複雑なリクエストや低速なネットワーク環境での不要なヒント表示が削減される。日常的な開発体験における小さなストレスを軽減するアップデートであり、すべてのユーザーに対してアップデートを推奨する。
