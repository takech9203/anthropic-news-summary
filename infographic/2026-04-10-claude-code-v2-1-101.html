2026-04-10

v2.1.98 -> v2.1.101

概要

/team-onboarding コマンド

🔄

--resume / --continue の信頼性大幅改善

大規模セッションでのコンテキスト喪失、チェーン回復エラー、クラッシュなど 7 件のセッション管理バグを修正

🔒

permissions.deny ルールが PreToolUse フックに正しく優先

フックの permissionDecision: "ask" が deny ルールをダウングレードする問題を修正。deny ルールが常に優先されるように

セキュリティ修正

⚠ POSIX which フォールバックにおけるコマンドインジェクション脆弱性が修正されました。LSP バイナリ検出を使用する全ユーザーに速やかなアップデートを推奨します。
1

POSIX which フォールバックのコマンドインジェクション修正

悪意のある入力を含むバイナリ名が which コマンドに渡された場合、任意のコマンドが実行される可能性がありました。修正により入力のサニタイズが適切に行われるようになりました

主要機能

👥 /team-onboarding

ローカルの Claude Code 使用状況からチームメンバー向けオンボーディングガイドを自動生成。新メンバーの導入効率を向上

🔐 OS CA 証明書

OS の CA 証明書ストアをデフォルトで信頼し、エンタープライズ TLS プロキシ環境での追加設定を不要化

☁ /ultraplan 自動環境作成

Web セットアップを事前に行わなくても、デフォルトのクラウド環境を自動作成するように改善

📝 Brief / Focus モード改善

Brief モードでプレーンテキスト応答時に構造化メッセージとして 1 回再試行。Focus モードはより自己完結的なサマリーを生成

⏱ エラーメッセージ改善

レートリミット時にどのリミットに達したか、いつリセットされるかを表示。拒否エラーに API の説明を含める

Before / After 比較

Before -- v2.1.98

  • TLS プロキシ環境で手動 CA 設定が必要
  • permissions.deny が hook の ask にダウングレード
  • 5 分ハードコードタイムアウト
  • --resume でコンテキスト喪失

After -- v2.1.101

  • OS CA ストアをデフォルトで信頼
  • deny ルールが常に優先
  • API_TIMEOUT_MS が正しく動作
  • セッション復元の信頼性大幅改善

改善の詳細 -- 13 件

☁

/ultraplan のクラウド環境自動作成

Web セットアップを事前に行わなくても、デフォルトのクラウド環境を自動作成するように改善

📝

Brief モードの再試行 / Focus モードのサマリー改善

Brief モードでプレーンテキスト応答時に構造化メッセージとして再試行。Focus モードではユーザーが最終メッセージのみ閲覧することを認識し、より自己完結的なサマリーを生成

⏱

レートリミット / 拒否エラーメッセージの改善

不透明な秒数カウントダウンの代わりに、どのリミットに達したか、いつリセットされるかを表示。拒否エラーに API の説明を含める

🔧

SDK query() のリソースクリーンアップ

for await から break した場合や await using を使用した場合に、サブプロセスと一時ファイルが適切にクリーンアップ

⚙

設定ファイルの耐障害性向上 / OTEL トレーシング改善

settings.json 内の認識されないフックイベント名があってもファイル全体が無視されなくなりました。OTEL でオプトインしない限り機密性の高いスパン属性が出力されないように改善

TypeScript SDK query() のリソースクリーンアップ 
import { query } from "@anthropic-ai/claude-code";

// for await から break した場合、リソースが自動クリーンアップされる
for await (const message of query({ prompt: "質問内容" })) {
  if (message.type === "result") {
    console.log(message.result);
    break; // サブプロセスと一時ファイルが自動的にクリーンアップされる
  }
}

// await using を使用した場合も同様
{
  await using session = query({ prompt: "質問内容" });
  // スコープ終了時に自動クリーンアップ
}

ユースケース

🏢 エンタープライズ環境での TLS プロキシ対応

TLS インスペクションプロキシを使用するエンタープライズ環境では、これまで Claude Code が接続エラーを起こすケースがありました。v2.1.101 では OS CA 証明書ストアをデフォルトで信頼するため、追加設定なしで動作します。Bedrock SigV4 認証の 403 エラー修正も含まれ、AWS 環境での利用が安定化

追加設定不要で動作

👥 チーム導入と /team-onboarding の活用

新しい /team-onboarding コマンドにより、ローカルの使用状況に基づいたオンボーディングガイドを自動生成できます。チームリーダーがガイドを作成し、新メンバーに共有することで、Claude Code の導入コストを大幅に削減

/team-onboarding を試す

⏰ 長時間開発セッションの安定性向上

メモリリーク修正、--resume / --continue の 7 件のバグ修正、レンダリング問題の修正により、長時間セッションの安定性が大幅に改善。5 分ハードコードタイムアウトの修正で、ローカル LLM や低速ゲートウェイでの利用も改善

安定性向上

技術的な詳細

1

OS CA 証明書ストアのデフォルト信頼

エンタープライズ TLS インスペクションプロキシが一般的な環境で、プロキシの CA 証明書を OS ストアから自動参照。CLAUDE_CODE_CERT_STORE=bundled で従来のバンドル CA のみの動作に復帰可能

2

permissions.deny ルールの優先度修正

PreToolUse フックが permissionDecision: "ask" を返した場合でも、permissions.deny に設定されたルールが常に優先。管理者が明示的に拒否したツール操作がユーザーの承認で実行されるリスクを解消

3

Bedrock SigV4 認証の修正

ANTHROPIC_AUTH_TOKEN、apiKeyHelper、ANTHROPIC_CUSTOM_HEADERS で Authorization ヘッダーが設定されている場合に SigV4 署名と競合して 403 エラーが発生する問題を修正

4

SDK query() のリソースクリーンアップ

for await ループからの break や await using 構文使用時に、サブプロセスと一時ファイルが適切にクリーンアップされるように修正。長時間稼働アプリケーションのリソースリークを防止

5

メモリリークとセッション復元の修正

仮想スクローラーのメッセージ履歴コピー蓄積によるメモリリーク修正。--resume / --continue の行き止まりブランチアンカー、サブエージェントブリッジ、file_path 欠落クラッシュなど複数の問題を修正

開発者への影響とアクション

🛡 全ての Claude Code ユーザー

POSIX which フォールバックのコマンドインジェクション脆弱性修正が含まれるため、速やかなアップデートを推奨。permissions.deny ルールの優先度修正も適用されます

速やかなアップデート推奨

🏢 エンタープライズ環境のユーザー

OS CA 証明書ストアのデフォルト信頼により、TLS インスペクションプロキシ環境での追加設定が不要に。/team-onboarding コマンドでチーム導入も効率化

TLS プロキシ設定を確認

☁ AWS Bedrock ユーザー

SigV4 認証の 403 エラー修正とカスタムキーバインドの読み込み問題修正の恩恵を受けます。ANTHROPIC_AUTH_TOKEN や ANTHROPIC_CUSTOM_HEADERS を使用している場合に特に影響

認証設定を確認

🔄 --resume / --continue ユーザー

セッション復元の信頼性が大幅に改善。メモリリーク修正と合わせて、長時間開発セッションの安定性が向上

安定性向上を確認

🔧 SDK 利用者 / プラグイン開発者

query() のリソースリーク防止、サブエージェントの MCP ツール継承修正、プラグインの複数問題修正が適用。長時間稼働アプリケーションの安定性が向上

リソース管理を確認

📋 設定ファイルに関する注意

settings.json に認識されないフックイベント名が含まれていた場合、ファイル全体が無視されなくなりました。アップデート後に設定が正しく適用されているか確認してください。

⚠ permissions.deny ルール使用者への注意

PreToolUse フックとの併用時に permissions.deny ルールが ask にダウングレードされていた可能性があります。修正後は deny ルールが正しく優先されるため、意図通りの動作になっているか確認してください。
